What is Control-M?

Control-M is BMC Software's enterprise workload automation platform — a single point of control for scheduling, running, and monitoring every critical business process across your entire IT estate.

The Central Idea

Every organization depends on processes running reliably and on time — salaries computed and distributed, databases backed up and restarted, reports generated, data pipelines fed. Control-M automates these processes, called jobs, in an organized, dependency-aware way.

A job is an execution unit — a script, command, or part of an external application. Jobs run in sequence forming a workflow. Control-M manages those workflows across every platform, application, and infrastructure your organization uses.

Core Concepts

How Control-M Works

Understanding the building blocks — from individual jobs to enterprise-wide orchestration.

🔧

Jobs

The atomic unit. A job is a script, OS command, or part of an external application (Hadoop, SAP, Snowflake). Each job carries its own scheduling criteria, dependencies, and post-processing actions — what to do after it succeeds or fails.

📁

Folders & SMART Folders

Jobs with related schedules are grouped into folders. A SMART Folder passes its scheduling, prerequisites, and actions down to all child jobs — so you define rules once at the folder level instead of per job, dramatically reducing configuration effort.

🔀

Flows & Dependencies

A Flow defines order dependency between jobs — Job B only starts after Job A ends successfully. One job can belong to multiple flows: Job 3 executes if either Job 1 or Job 2 completes. This creates complex, conditional pipelines without custom scripting.

📅

Calendars & Scheduling

Rule-Based Calendars define when jobs run — weekdays, month-end, specific dates. Confirmation Calendars automatically handle exceptions like public holidays. If a scheduled day falls on a holiday, Control-M applies your policy (skip, run next day, run before) without manual intervention.

🖥️

Agents

Control-M Agents are installed on the hosts where your jobs actually run. The Agent receives job instructions from the Server, executes the job, monitors it, and reports status back. Agentless execution via Remote Hosts is also supported for environments where installing software isn't possible.

🔐

Resources & Lock Control

Lock Resources prevent jobs from accessing the same file, host, or device simultaneously. Resource Pools cap total CPU or memory usage so no workflow can starve others. This prevents system overload even at peak volume of 100,000 executions per server per day.

Architecture

Three-Layer Architecture

Control-M uses a distributed microservices architecture built on Apache Kafka and Zookeeper for high-throughput, fault-tolerant orchestration.

🏛️

Enterprise Manager (EM)

The central console. Hosts all client tools — the Planning UI, Configuration Manager, Self-Service portal, Workload Change Manager, and the Automation API for DevOps. The GUI Server handles client communication with load-balancing. A Global Conditions Server distributes events for cross-server dependencies.

⚙️

Control-M Server

The scheduling engine. Manages job scheduling, workflow coordination, and workload balancing. Maintains a local Active Jobs Database. Server-side microservices include an API Gateway, Job Info Service, Job Order Service, and Scheduling Service — all synchronized via Kafka messaging.

🤖

Agents & Remote Hosts

Installed on every target machine where jobs run. Receives job definitions from the Server, executes them, and reports status back. Supports counters, multiple notification types, and plug-ins for specific applications. Remote Hosts enable agentless execution where direct installation isn't possible.

Monitoring & Alerting

Real-Time Operational Control

The Monitoring domain gives you live visibility into every job and workflow. Investigate issues through three tabs: Log (every status change with timestamps), Output (job result and exit codes), and Waiting Info (what events or resources a job is blocked on).

Alerts are classified as Normal, Urgent, or Critical. They can auto-open tickets in Jira, trigger scripts, or push to external monitoring tools like Dynatrace and Datadog via WebSocket streams.

Getting Started

How to Get Up and Running

Whether you're on SaaS or self-hosted, Control-M follows the same setup sequence. SaaS users skip Steps 1–2 entirely — infrastructure is handled by BMC.

1

Log In

Access Control-M via the web UI. SaaS users get credentials immediately; self-hosted requires server installation first.

2

Install Agents

Deploy Control-M Agents on every host where jobs will run. Agents handle execution and status reporting back to the Server.

3

Define Connections

Configure Connection Profiles for each plug-in (Snowflake, SAP, Hadoop, etc.) so Control-M can authenticate and submit jobs to those systems.

4

Build Workflows

Create a Workspace in the Planning domain. Add a SMART Folder, drag in jobs, define scheduling rules and dependencies, then check in to the database.

5

Monitor & Iterate

Use the Monitoring domain to watch live job statuses, review logs and output, configure alerts, generate reports, and troubleshoot failures.

📖 Key Terminology

Job

A single execution unit — script, OS command, or app-level task (Hadoop, SAP, Snowflake plug-in)

SMART Folder

A container whose scheduling, prerequisites, and actions are inherited by all child jobs automatically

Flow

An order-dependency definition: Job B only runs if Job A ends successfully

New Day

Daily reset cycle (default midnight) when completed folders are cleared and next-day jobs are ordered

Run Queue

The set of jobs scheduled to execute today, in Waiting or Executing state

SLA Job

Maps the critical path of a workflow that must complete within a user-defined time window

Definition

What is AI Workflow Orchestration?

AI workflow orchestration acts as the control system for complex AI environments — coordinating data pipelines, model execution, dependencies, triggers, and error handling to ensure reliable operations.

Running AI in production can be like managing air traffic in unpredictable skies. Data arrives continuously, some systems act autonomously, and models trigger in real time — requiring precise coordination to avoid delays or failures. As AI systems evolve, agentic orchestration introduces autonomous decision-making, making reliable execution even more critical.

How It Works

A Typical Big Data Workflow

1

Data arrives in S3 / Azure Data Lake (event trigger)

Control-M detects the file event and starts the workflow automatically — no polling required.

2

Spark job runs on Databricks / EMR (transformation)

Control-M submits the Spark job, monitors cluster status, and waits for successful completion before proceeding.

3

Data quality check (validation job)

A custom script or Great Expectations check verifies row counts, null thresholds, and schema. On failure, an alert fires and the pipeline halts — not the whole environment.

4

Load to Snowflake (COPY INTO)

Control-M runs the Snowflake load job, confirms row counts match, and updates an event condition.

5

Downstream jobs trigger (BI refresh, reports, ML)

The OUT condition from step 4 releases dependent jobs across other SMART folders — BI dashboards refresh, scheduled reports generate, ML model scores update.

What It Is

Workflow Automation, Built Like Software

Traditionally, job scheduling was done through GUIs — click-and-configure workflows that lived only inside the scheduler. Jobs-as-Code changes that: you define your jobs and workflows in JSON or Python code, store them in Git, test them like software, and deploy them through your existing CI/CD pipeline.

This means your automation is versioned, reviewable, and repeatable. A developer can add a new job to a workflow, open a pull request, get it reviewed, and deploy it to production — exactly like any other code change.

📝

Code in Any Editor

Write job definitions in JSON or Python using any text editor or IDE. No GUI required. Jobs live alongside application code in the same repository.

🧪

Automated Testing

Run jobs against a local Control-M Workbench environment to test before deploying. Use your existing testing frameworks — pytest, JUnit, anything.

🚀

CI/CD Deployment

Deploy job definitions through Jenkins, GitLab CI, GitHub Actions, or any CI/CD system — promoting automatically from dev to staging to production.

How It Works

The Jobs-as-Code Pipeline

1️⃣

Build

Developer writes job definitions in JSON/Python and commits to Git. The Automation API CLI validates syntax and structure locally.

2️⃣

Test

CI pipeline deploys jobs to a Control-M Workbench (local dev environment), runs them, and asserts expected outputs — just like unit tests for your application.

3️⃣

Deploy

On merge to main, the pipeline deploys the updated job definitions to staging then production Control-M — no manual GUI work required.

Example — A Simple Job Definition in JSON

{
  "FinanceWorkflow": {
    "Type": "Folder",
    "ControlmServer": "prod-server",
    "When": { "Weekdays": ["MON","TUE","WED","THU","FRI"] },

    "ExtractData": {
      "Type": "Job:Script",
      "FileName": "extract.sh",
      "FilePath": "/jobs/finance",
      "RunAs": "svc-finance"
    },
    "TransformData": {
      "Type": "Job:Script",
      "FileName": "transform.py",
      "FilePath": "/jobs/finance",
      "RunAs": "svc-finance"
    },
    "LoadToSnowflake": {
      "Type": "Job:Database:Snowflake",
      "ConnectionProfile": "snowflake-prod",
      "SQL": "COPY INTO finance_dw FROM @stage/finance/"
    },
    "Flow": {
      "Type": "Flow",
      "Sequence": ["ExtractData", "TransformData", "LoadToSnowflake"]
    }
  }
}

Migration

Migrating from SAP Job Scheduler to Control-M

Control-M includes conversion tools that import existing SAP batch job definitions and translate them into Control-M SMART Folders automatically — no manual re-entry. The process:

1

Export existing SAP job definitions

Use Control-M's SAP conversion utility to extract job schedules from SM37 or BTP Job Scheduler.

2

Auto-convert to Control-M SMART Folders

The conversion tool maps SAP scheduling attributes to Control-M equivalents — calendars, dependencies, parameters, and post-processing actions.

3

Run both systems in parallel

Validate Control-M schedules alongside existing SAP schedules before cutting over — no risk to production during migration.

4

Decommission SAP scheduler

Once validated, disable SAP's native scheduler and manage all SAP jobs through Control-M — unified with the rest of your enterprise automation.
